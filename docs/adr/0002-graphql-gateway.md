## Context

В проекте **Auth Service** GraphQL используется как основной API gateway между фронтендом и backend сервисами.  

Основные требования:

- Унифицированный интерфейс для всех команд и запросов.  
- Поддержка cookie-based сессий (`Set-Cookie`) и кросс-доменных запросов.  
- Масштабируемость и возможность интеграции новых сервисов в будущем.  

---

## Decision

Решено использовать **GraphQL Gateway** вместо прямого REST или gRPC по следующим причинам:

### 1. Единый API для команд и запросов

- Все операции проходят через один GraphQL endpoint, что упрощает маршрутизацию и обработку ошибок.  
- Позволяет фронтенду взаимодействовать с одной точкой входа, без необходимости множества REST endpoints.

### 2. Backward Compatibility

- Любые будущие изменения backend сервисов можно интегрировать **без изменения фронтенда**.  
- Новые поля и типы добавляются как необязательные, не ломая существующих клиентов.

### 3. Cookie-based session support

- GraphQL позволяет передавать cookie с флагом `credentials: include`.  
- Упрощает управление сессией на frontend и сохраняет безопасность (HttpOnly, Secure, SameSite).  

---

## Trade-offs

| Trade-off | Обоснование |
|-----------|------------|
| Не REST | REST потребовал бы множество endpoints для команд и запросов; GraphQL объединяет их в одной схеме. |
| Не gRPC | Cookie-based сессии сложнее передавать через gRPC, требуется ручная обработка на фронтенде. |

---

## Consequences

- Унифицированный API, меньше дублирования кода.  
- Все команды и запросы проходят через единый gateway, легко масштабировать и логировать.  
- Cookie-based аутентификация поддерживается автоматически.  
- Возможность добавления новых сервисов без изменения клиента.
