## Context

Для проекта **Auth Service** необходимо реализовать систему аутентификации и управления сессиями для пользователей с возможностью регистрации, авторизации и восстановления пароля.  

Основные требования:

- Поддержка кросс-доменных запросов (CORS) и безопасная передача аутентификационных данных.
- Простота интеграции с фронтендом.
- Совместимость с будущими расширениями (OAuth2, OpenID, масштабирование нескольких сервисов).

---

## Decision

Решено использовать **cookie-based сессии**, а не JWT/OpenID, по следующим причинам:

### 1. Простота фронтенд интеграции

- Сессии через cookie автоматически отправляются браузером при каждом запросе с флагом `credentials: include`.  
- Нет необходимости вручную хранить токены на клиенте (localStorage, sessionStorage).  
- Обеспечивает **обратную совместимость** (backward compatibility) для существующих клиентов.

### 2. Безопасность

- Cookie можно настроить с флагами `HttpOnly`, `Secure`, `SameSite`, что снижает риск XSS и CSRF атак.  
- JWT требует отдельного хранения на фронтенде и управления обновлением (refresh tokens), что повышает сложность и потенциальные риски.

### 3. Масштабируемость и расширяемость

- Cookie упрощают масштабирование до нескольких сервисов при необходимости кросс-доменных запросов.  
- При интеграции OAuth2/OpenID для сервис-сервис аутентификации можно добавить JWT или PKCE, сохранив cookie для фронтенда.  

---

## Trade-offs

| Trade-off | Обоснование |
|-----------|------------|
| Не использован JWT | Сервис один, проще и безопаснее использовать cookie-based сессию для фронтенда. |
| Нет подтверждения пароля по почте | Планируется добавить в продакшн версии. |

---

## Consequences

- Frontend интеграция упрощена, нет необходимости управлять токенами вручную.  
- Безопасность обеспечена через стандартные cookie-флаги.  
- Архитектура легко масштабируется с учетом будущих интеграций OAuth2/OpenID.  
- Необходимо корректно настроить кросс-доменные cookie при работе с различными доменами.
